# Design: Embedding Model Mismatch Protection

## 1. Problem Statement

Embedding vectors are only comparable when generated by the same model. OpenVault stores embeddings per-chat (`chatMetadata.openvault`) but the embedding model setting (`embeddingSource`) is **global** — shared across all chats.

If a user:
1. Generates embeddings in Chat A with `multilingual-e5-small` (384d)
2. Switches to Chat B, changes model to `bge-small-en-v1.5` (384d) or `embeddinggemma-300m` (768d)
3. Returns to Chat A

...the system will perform cosine similarity between vectors from different latent spaces. This either:
- **Crashes** (dimension mismatch: 384 vs 768)
- **Returns garbage** (same dimensions but incompatible spaces)

Currently there is no record of *which model* generated the stored embeddings.

## 2. Goals & Non-Goals

### Must do
- Record which embedding model generated a chat's embeddings
- Detect model mismatch on chat load
- Automatically invalidate stale embeddings (memories, graph nodes, communities)
- Let the existing background worker re-embed transparently
- Remove the manual "Clear Embeddings" button (redundant with auto-system)

### Won't do
- Per-object model tagging (over-engineering for free local embeddings)
- Versioned embedding namespaces (bloats metadata, no user benefit)
- Support for paid API embeddings (Transformers.js only — revisit if Ollama becomes primary)
- UI lock/warning dialogs (frictionless auto-recompute is better UX)

## 3. Proposed Architecture: "Tag & Bulk Wipe"

### Flow

```
CHAT_CHANGED event fires
       │
       ▼
onChatChanged() runs
       │
       ▼
Compare: settings.embeddingSource vs data.embedding_model_id
       │
       ├── MATCH (or no embeddings exist) → no-op, worker continues normally
       │
       └── MISMATCH →
             1. Wipe all embedding_b64 from memories[], graph.nodes{}, communities{}
             2. Set data.embedding_model_id = settings.embeddingSource
             3. Clear in-memory embedding cache
             4. Save metadata
             5. Log: "Embedding model changed (was: X, now: Y). Cleared N embeddings."
             │
             ▼
       Worker wakes up, sees missing embeddings, re-generates them
```

### Why this works
- The background worker already checks `hasEmbedding()` and generates missing vectors
- Re-embedding is free (local Transformers.js) and runs on idle/yield
- Zero UI friction — user doesn't notice the switchover
- Single source of truth: one `embedding_model_id` at the data root

### Edge cases
| Scenario | Behavior |
|----------|----------|
| Brand new chat, no embeddings | `embedding_model_id` set on first embed batch. No wipe needed. |
| Chat with embeddings, same model | `embedding_model_id` matches → no-op |
| Chat with embeddings, different model | Bulk wipe → worker re-embeds |
| `embedding_model_id` missing (legacy chat) | Treat as mismatch if embeddings exist → wipe & tag. If no embeddings → just tag. |
| Model changed mid-worker-run | Worker checks `getSessionSignal()` every iteration. Chat switch triggers abort. On re-entry, mismatch detected → wipe. |
| Ollama selected but not configured | `isEmbeddingsEnabled()` returns false → worker skips. Embeddings stay wiped until model is valid. |

## 4. Data Model Changes

### New field in `chatMetadata.openvault`

```typescript
{
  embedding_model_id: string | undefined,  // e.g. "multilingual-e5-small", "bge-small-en-v1.5", "ollama"
  // ... existing fields unchanged
}
```

Single root-level field. No per-object changes.

### Affected embedded objects (wipe targets)
| Object | Path | Embedding field |
|--------|------|-----------------|
| Memory | `data.memories[i]` | `embedding_b64` (via `deleteEmbedding()`) |
| Graph node | `data.graph.nodes[key]` | `embedding_b64` (via `deleteEmbedding()`) |
| Community | `data.communities[key]` | `embedding_b64` (via `deleteEmbedding()`) |

## 5. Interface / API Design

### New function: `invalidateStaleEmbeddings(data, currentModelId)` in `src/utils/data.js`

```javascript
/**
 * Check if stored embeddings were generated by a different model.
 * If mismatch detected, wipe all embeddings and update the model tag.
 *
 * @param {Object} data - OpenVault chat data (chatMetadata.openvault)
 * @param {string} currentModelId - Currently selected embedding source
 * @returns {number} Number of embeddings wiped (0 = no mismatch)
 */
export function invalidateStaleEmbeddings(data, currentModelId) {
    // No embeddings exist yet → just stamp the model ID
    if (!data.embedding_model_id) {
        const hasAny = countEmbeddings(data) > 0;
        if (!hasAny) {
            data.embedding_model_id = currentModelId;
            return 0;
        }
        // Legacy chat with embeddings but no tag → treat as mismatch
    }

    // Match → nothing to do
    if (data.embedding_model_id === currentModelId) {
        return 0;
    }

    // MISMATCH: wipe everything
    const oldModel = data.embedding_model_id || 'unknown';
    let count = 0;

    // Memories
    for (const m of data.memories || []) {
        if (hasEmbedding(m)) { deleteEmbedding(m); count++; }
    }

    // Graph nodes
    for (const node of Object.values(data.graph?.nodes || {})) {
        if (hasEmbedding(node)) { deleteEmbedding(node); count++; }
    }

    // Communities
    for (const community of Object.values(data.communities || {})) {
        if (hasEmbedding(community)) { deleteEmbedding(community); count++; }
    }

    data.embedding_model_id = currentModelId;
    log(`Embedding model changed (${oldModel} → ${currentModelId}). Wiped ${count} embeddings.`);
    return count;
}
```

### Call site: `onChatChanged()` in `src/events.js`

```javascript
// After getOpenVaultData(), before worker wake:
const settings = deps.getExtensionSettings()[extensionName];
if (data && settings?.embeddingSource) {
    const wiped = invalidateStaleEmbeddings(data, settings.embeddingSource);
    if (wiped > 0) {
        await saveOpenVaultData();
    }
}
```

### Also stamp on embed: `enrichEventsWithEmbeddings()` / `generateEmbeddingsForMemories()`

After successfully embedding a batch, ensure `data.embedding_model_id` is set if it wasn't already. This handles the "first embed in a new chat" case without requiring a separate code path.

### UI removal

- Delete `#openvault_delete_embeddings_btn` from `templates/settings_panel.html`
- Delete `handleDeleteEmbeddings()` from `src/ui/settings.js`
- Delete click binding `$('#openvault_delete_embeddings_btn').on('click', ...)`
- Keep `deleteCurrentChatEmbeddings()` in `data.js` — still useful internally and for tests

## 6. Risks & Edge Cases

| Risk | Mitigation |
|------|------------|
| **Legacy chats with no `embedding_model_id`** | If embeddings exist but tag is missing, treat as mismatch → wipe. One-time cost, then tagged going forward. |
| **Worker re-embeds large chat (1000+ memories)** | Worker already processes in batches of 5 with `yieldToMain`. May take a few minutes. User can keep chatting. |
| **Race: model setting changed while worker is embedding** | Worker checks `getSessionSignal()` each iteration. If user changes model without switching chats, the next `onChatChanged` or manual action catches it. For within-same-chat changes: the `embeddingSource` change handler in settings.js should also call `invalidateStaleEmbeddings`. |
| **Ollama model name change** | Ollama's `embeddingSource` is just `"ollama"` — changing the Ollama model name behind it won't trigger mismatch. Acceptable trade-off: Ollama is not the primary target. |
| **Graph nodes re-embedded during merge** | `mergeOrInsertEntity` already calls `setEmbedding` — wiped nodes get re-embedded during next extraction cycle. |
