# Design: Dynamic Concurrency (Ladder Queue) for Phase 2

## 1. Problem Statement

Currently, Phase 2 Enrichment operations in OpenVault (Reflections, Community Summaries, Edge Consolidation, and Global Synthesis) process sequentially using a standard `for...of` loop with `await`. 

When users trigger a heavy backfill or reach a major 50-message boundary, this sequential processing causes significant bottlenecks. Cloud API providers (like Kimi, Qwen, or OpenAI) can handle parallel requests, but aggressively spamming them via `Promise.all()` triggers `429 Too Many Requests` or network timeouts.

We need a dynamic concurrency system that:
1. Optimistically processes Phase 2 LLM requests in parallel.
2. Automatically halves its concurrency (Drops the Ladder) upon detecting rate limits or timeouts.
3. Gradually increases concurrency back to the maximum limit (Climbs the Ladder) upon subsequent successes.

## 2. Goals & Non-Goals

### Must Do
- Import `p-queue` dynamically via `src/utils/cdn.js`.
- Create a lightweight AIMD wrapper around `p-queue` (`src/utils/queue.js`).
- Refactor loops in `src/reflection/reflect.js` and `src/graph/communities.js` to utilize the queue.
- Add a `maxConcurrency` user setting (defaulting to `1` to protect local LLM users).
- Ensure the ladder automatically pauses for 3-5 seconds on a 429/timeout before resuming.
- Add `p-queue` to `package.json` and alias it in `vitest.config.js` for testing.

### Won't Do
- **Do NOT parallelize Phase 1 (Event -> Graph extraction).** The Graph extraction prompt explicitly requires the structured output from the Event extraction. They are fundamentally sequential.
- Do not implement complex retry-logic inside the queue wrapper itself (the underlying functions already have `try/catch` and fallback logic).
- Do not override the user's `maxConcurrency` setting; the ladder can only scale *down* from the user's ceiling, never above it.

## 3. Proposed Architecture

### 3.1 The AIMD Ladder Queue (`src/utils/queue.js`)

We will introduce a new utility file that initializes `p-queue` and attaches the Additive Increase, Multiplicative Decrease (AIMD) logic.

```javascript
import { cdnImport } from './cdn.js';
import { logDebug, logWarn } from './logging.js';

let PQueue;

/**
 * Creates an AIMD-governed task queue.
 * @param {number} maxConcurrency - The absolute ceiling for parallel tasks.
 */
export async function createLadderQueue(maxConcurrency) {
    if (!PQueue) {
        const module = await cdnImport('p-queue');
        PQueue = module.default;
    }

    const queue = new PQueue({ concurrency: maxConcurrency });
    let currentLimit = maxConcurrency;

    const add = async (taskFn) => {
        return queue.add(async () => {
            try {
                const result = await taskFn();
                
                // Additive Increase: Slowly climb back up on success
                if (currentLimit < maxConcurrency) {
                    currentLimit = Math.min(maxConcurrency, currentLimit + 0.5);
                    queue.concurrency = Math.floor(currentLimit);
                }
                return result;

            } catch (error) {
                const isRateLimit = 
                    error.message?.includes('429') || 
                    error.message?.includes('timeout') || 
                    error.status === 429;

                if (isRateLimit) {
                    // Multiplicative Decrease: Drop the ladder
                    currentLimit = Math.max(1, Math.floor(currentLimit / 2));
                    queue.concurrency = Math.floor(currentLimit);
                    
                    logWarn(`Rate limit hit. Dropping concurrency to ${queue.concurrency}`);

                    // Pause queue to let the API breathe
                    if (!queue.isPaused) {
                        queue.pause();
                        setTimeout(() => {
                            logDebug('Resuming ladder queue');
                            queue.start();
                        }, 4000); // 4 second cool-off
                    }
                }
                throw error; 
            }
        });
    };

    return { 
        add, 
        onIdle: () => queue.onIdle(),
        get concurrency() { return queue.concurrency; }
    };
}
```

### 3.2 Refactoring Execution Loops

We will replace the sequential `for...of` loops in Phase 2 with mapped queue additions. Because the internal logic already handles `try/catch`, we can safely `Promise.all` the queue array.

**Target 1: Community Summarization (`src/graph/communities.js`)**
```javascript
// Old:
for (const [communityId, group] of Object.entries(communityGroups)) {
    // await callLLM(...)
}

// New:
const { createLadderQueue } = await import('../utils/queue.js');
const ladderQueue = await createLadderQueue(settings.maxConcurrency);
const promises = [];

for (const [communityId, group] of Object.entries(communityGroups)) {
    promises.push(ladderQueue.add(async () => {
        // existing community summarization try/catch block
    }));
}
await Promise.all(promises);
```

**Target 2: Reflections (`src/reflection/reflect.js`)**
```javascript
const ladderQueue = await createLadderQueue(settings.maxConcurrency);
const promises = [];

for (const characterName of characterNames) {
    if (shouldReflect(data.reflection_state, characterName, reflectionThreshold)) {
        promises.push(ladderQueue.add(async () => {
            // existing generateReflections logic
        }));
    }
}
await Promise.all(promises);
```

**Target 3: Edge Consolidation (`src/graph/graph.js`)**
Applied exactly the same way to `consolidateEdges`, ensuring that edge description synthesis operates in parallel.

## 6. Risks & Edge Cases

### Risk: VRAM Crashes for Local LLM Users
- **Scenario:** A user using Ollama sets concurrency to 5, instantly overloading their VRAM with parallel context windows, crashing the backend.
- **Mitigation:** The setting defaults to `1`. The UI tooltip explicitly warns against raising it unless using a cloud API.

### Risk: Premature Optimization in Phase 1
- **Scenario:** A developer attempts to reuse `createLadderQueue` for `extractMemories` (Events/Graph).
- **Mitigation:** `ARCHITECTURE.md` and inline comments in `extract.js` will explicitly document that Stage A and Stage B are functionally dependent and mathematically cannot be parallelized.

### Risk: Promise Rejection Tearing Down the Batch
- **Scenario:** One community summary fails inside the `Promise.all` array, crashing the entire Phase 2 pipeline.
- **Mitigation:** Existing codebase already wraps the interior of these loops in isolated `try/catch` blocks that log the error and `continue`. The Promises returned to `ladderQueue.add()` will cleanly resolve even if the LLM call fails, preventing `Promise.all` from rejecting. (Note: The `isRateLimit` catch in the wrapper re-throws the error *only* to ensure the local `try/catch` logs the failure, but it won't crash the queue).
