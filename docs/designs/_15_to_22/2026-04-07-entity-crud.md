# Entity CRUD: Edit, Delete, and Alias Management

Phase 1 of the Vault Editing Suite. Gives users direct control over entity data — editing fields, managing aliases, and deleting bad entries.

## Problem

Entities are auto-generated by the extraction pipeline and cannot be edited by the user. If the LLM hallucinates entity details, assigns the wrong type, or creates a duplicate under a different name, there is no way to fix it. The only recourse is to delete the entire chat's data and re-extract.

The reviewer also identified the "masked figure" problem: an entity introduced under one name and later revealed as another has no way to be manually linked until the automatic merge catches up (which it often doesn't).

## Solution

Make entity cards editable with inline editing (matching the existing memory edit pattern), add alias management, add entity deletion with edge cleanup, and restructure the World tab into two focused tabs.

## Tab Restructure

Replace the single "World" tab with two tabs:

| Tab | `data-tab` | Content |
|---|---|---|
| Entities | `entities` | Editable entity cards with search, filter, CRUD, aliases |
| Communities | `communities` | Community summaries (read-only for now; editable in Phase 3) |

The Communities tab is a straight move of the existing community section HTML from the World tab into its own tab content div. No functional changes.

## Entity Rename — Edge Rewriting

Renaming an entity changes its key in the graph. Since edges are keyed as `sourceKey__targetKey`, all edges referencing the old key must be rewritten.

Algorithm in `updateEntity()`:

```
1. Normalize the new name → newKey
2. If newKey === oldKey: simple field update (no edge rewrite needed)
3. If newKey !== oldKey:
   a. **Collision check**: if graphData.nodes[newKey] already exists,
      abort and return null. Show toast:
      "An entity named '{newName}' already exists. Merging is coming in a future update."
   b. Create graphData.nodes[newKey] = copy of old node with updated fields
   c. Delete graphData.nodes[oldKey]
   d. For each edge in graphData.edges:
      - If edge.source === oldKey: rewrite to newKey
      - If edge.target === oldKey: rewrite to newKey
      - Delete old edge key, insert under new composite key
   e. Set graphData._mergeRedirects[oldKey] = newKey
   f. Call deleteEmbedding(node) — clears embedding, embedding_b64, _st_synced
4. Save
```

**Collision blocker:** Phase 1 blocks renames to existing entity names. This prevents silently overwriting an existing node. Entity merging (Phase 2) will handle this case properly.

The `_mergeRedirects` entry ensures that any future extraction output referencing the old name resolves to the new key via the existing `_resolveKey()` function. `deleteEmbedding()` (from `src/utils/embedding-codec.js`) clears `embedding`, `embedding_b64`, and `_st_synced`, ensuring the vector store re-syncs on the next cycle.

## Entity Deletion

Triggered by the delete button on the view-mode card.

Flow:
1. Count edges for this entity (`edges.filter(e => e.source === key || e.target === key).length`)
2. `confirm('Delete "{name}"? This will also remove ${edgeCount} connected relationship(s).')` — browser confirm dialog with edge count
3. Call `deleteEntity(key)` from `src/store/chat-data.js`
3. Implementation:
   - Delete `graphData.nodes[key]`
   - Scan `graphData.edges`, remove any edge where `source === key` or `target === key`
   - Scan `graphData._mergeRedirects`, remove any entry where value === key or key === key
4. Save
5. Re-render entity list

No undo. The confirm dialog is the safety net, matching the existing memory delete pattern.

## Alias Management

Aliases are stored in `node.aliases` (array of strings, optional field in `GraphNodeSchema`).

### Add Alias

User types into the alias input field and clicks "Add" (or presses Enter):

1. Trim whitespace, skip if empty
2. Check that the alias doesn't already exist in `node.aliases` (case-insensitive)
3. Push to `node.aliases`
4. Re-render the alias chip list in the edit form (not a full save — aliases are saved when the user clicks Save)

### Remove Alias

User clicks × on an alias chip:

1. Remove from `node.aliases` array
2. Re-render the chip list

No guard against removing all aliases — the entity name itself is the primary identifier, aliases are supplementary.

## What is NOT Changed

- **Extraction pipeline** — Entity extraction, graph building, merge logic: untouched
- **Retrieval pipeline** — Scoring, injection, world context: untouched
- **Entity merge logic** — Automatic semantic merge still runs; manual editing is additive
- **Community summaries** — Moved to own tab but still read-only (Phase 3)
- **Edges display** — Not in this phase; Phase 4 adds edge viewing per entity
- **Schema** — No schema changes. The `aliases` field already exists in `GraphNodeSchema`
